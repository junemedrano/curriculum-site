12-Week Programmer Syllabus

A practical 12-week program for aspiring software engineers, CS students, and game developers. Each week covers real skills, project work, and professional tools — differentiated by track so students are never doing generic exercises.

Updated 2025 · SWE · CS · Game Dev
Three tracks run in parallel through the same weekly topics. SWE emphasizes production habits and team workflow. CS emphasizes theory, proofs, and algorithmic rigor. Game Dev emphasizes Unity, C#, and shipping interactive projects. Math focus and reading recommendations are track-specific throughout.

How This Program Works

The 12-week structure moves from foundations to deployment — Git and environment setup, through data structures, testing, web and API basics, databases, and into a final project you can actually show. Each week has one shared topic block and three differentiated columns showing how that topic looks in your track.

Math is not treated as background material. Each week identifies the specific mathematical concepts that underpin what you're building — not to prepare for an exam, but because understanding the underlying structure is what separates a programmer who can follow a tutorial from one who can debug an unfamiliar system. The difficulty and emphasis scales by track: CS students encounter proofs and formal definitions; SWE students encounter applied reasoning and complexity; Game Dev students encounter the geometry and linear algebra that shows up directly in their engines.

Recommended reading from the main bibliography is listed at the end of each section. These are not assigned — they're the books that answer the question "where do I go deeper on this" if a week's topic opens a door you want to walk through.

Week 1 — Foundations & Git

Environment setup, version control, and the professional workflow that everything else depends on.

W1 Setup, Git & IDE Basics topics: Git clone · commit · push · branching · IDE · project structure
SWE
Collaborative Git workflow, branching strategies (Gitflow vs. trunk-based), pull request culture. Set up a real project structure from day one — no single-file scripts.
CS
Version control as a formal system: commit graphs as DAGs, branching as tree operations. Introduction to reproducibility and documentation discipline in academic work.
Game Dev
Unity project setup with Git LFS for large binary assets. Scene management, .gitignore for Unity, avoiding merge conflicts on .unity files.
Math focus — SWE: Sets and functions (inputs, outputs, side effects). Thinking about state as a mapping. · CS: Directed acyclic graphs — commit history is a DAG. Introduction to graph vocabulary: nodes, edges, reachability. · Game Dev: File system as a tree. Basic graph intuition for scene hierarchies.

Week 2 — Core Programming Concepts

Variables, functions, and control flow — plus the mathematical structures that make these concepts precise.

W2 Variables, Functions & Control Flow topics: data types · loops · conditionals · functions · debugging
SWE
Emphasis on modular, testable code from the start. Functions as units of behavior. Introduction to pure vs. impure functions and why that matters for testing.
CS
Functions as mathematical objects. Domain, codomain, and range. Introduction to Boolean algebra: conditionals as logical expressions. Loop invariants as informal proofs.
Game Dev
C# scripting in Unity: MonoBehaviour lifecycle (Start, Update, FixedUpdate). Conditionals for game state. Debugging in the Unity editor.
Math focus — SWE: Functions as input-output contracts. Preconditions and postconditions. · CS: Propositional logic (AND, OR, NOT, implication) as the foundation of conditionals. Truth tables. Loop invariants. · Game Dev: Frame-rate math: Update runs at variable dt; FixedUpdate at fixed intervals. Basic time arithmetic.

Week 3 — Data Structures & OOP

Collections, objects, and the abstract structures that underpin both.

W3 OOP, Arrays, Lists, Dictionaries topics: classes · objects · methods · arrays · lists · hash maps
SWE
OOP design patterns for small applications. Unit tests for objects. Introduction to encapsulation and why it reduces surface area for bugs.
CS
Data structure complexity: array access O(1), list insertion O(n), hash map average O(1). Memory layout. Introduction to abstract data types vs. concrete implementations.
Game Dev
Unity components as OOP in practice. ScriptableObjects for data. Managing collections of game objects (lists of enemies, inventory items).
Math focus — SWE: Invariants — what properties does a class guarantee about its state at all times? · CS: Big-O notation. Recurrence relations for recursive structures. Hash functions as mathematical mappings. · Game Dev: Spatial partitioning intuition: why a list of 500 enemies and a quadtree behave differently under load.

Week 4 — Git Advanced & Collaboration

Feature branches, pull requests, conflict resolution, and team workflow patterns.

W4 Branches, Pull Requests & Conflict Resolution topics: feature branches · PRs · merging · rebasing · Git workflows
SWE
Team project with proper code review workflow. Branch naming conventions, PR templates, reviewer comments. Introduction to CI as a quality gate on PRs.
CS
Collaborative coding and maintaining reproducibility. Merge as a graph operation: three-way merge as finding the lowest common ancestor in the commit DAG.
Game Dev
Binary asset versioning with Git LFS. Avoiding scene file conflicts through prefab-first design. Branching for feature isolation in game projects.
Math focus — SWE / CS: Merge algorithms as graph problems — lowest common ancestor, three-way merge. Conflict as divergent state; resolution as a reconciliation function. · Game Dev: State divergence — two branches are two versions of the same game state. Conflict is state overlap that requires human resolution.

Week 5 — Error Handling & Testing

Exceptions, assertions, unit tests, and the discipline of knowing whether your code is correct.

W5 Error Handling & Unit Testing topics: exceptions · assertions · test-driven development · debugging tools
SWE
Test-driven development workflow: red-green-refactor. CI basics — tests that run on every commit. Difference between unit, integration, and end-to-end tests.
CS
Formal correctness: preconditions, postconditions, and invariants. Assertions as executable specifications. Introduction to the relationship between testing and proof.
Game Dev
Unity Test Framework for scripts and scenes. Testing game logic in isolation from the Unity engine. Debugging with breakpoints and the Unity profiler.
Math focus — SWE: Coverage as a set problem: test cases as a subset of the input space. What fraction of inputs are you actually testing? · CS: Hoare triples ({P} C {Q}) — precondition, command, postcondition as a formal model of what a test asserts. · Game Dev: Edge cases as boundary conditions — testing the endpoints of numeric ranges (health = 0, health = max).

Week 7 — Databases

Persistence, relational models, querying, and when to use SQL vs. not.

W7 SQL & NoSQL Basics topics: relational DBs · PostgreSQL · MongoDB · schema design · CRUD
SWE
Schema design, normalization to 3NF, writing real queries, and connecting to backend services. Introduction to ORMs and when they hide things you need to see.
CS
Relational algebra as the mathematical foundation of SQL. Set operations: SELECT as projection, WHERE as selection, JOIN as product. Normal forms as mathematical constraints on relations.
Game Dev
Persisting player progress, stats, and inventory. SQLite for local storage; cloud-backed NoSQL (Firebase/Firestore) for live data. Serialization patterns for save systems.
Math focus — SWE: ACID properties as mathematical guarantees: atomicity (all or nothing), consistency (invariants hold), isolation (serializability), durability (persistence). · CS: Relational algebra: sets, tuples, projection, selection, Cartesian product, join. SQL as a notation for relational algebra expressions. · Game Dev: Entity-component data modeling — how a game's data model maps to relational or document structures.

Week 8 — Advanced JS & Type Safety

ES6+ features, TypeScript fundamentals, and why types are a tool for reasoning rather than just a compiler requirement.

W8 ES6+ Features & TypeScript Introduction topics: ES6 modules · closures · destructuring · TS types · generics

A type-correct program is a proof that it won't produce certain classes of error. · Game Dev: Type safety in component systems: why Unity's GetComponent<T>() is generics in practice.

After 12 Weeks

The program covers the practical foundation. What it doesn't cover is everything that comes next — and the honest answer is that what comes next depends entirely on your track.

For SWE students: systems design, distributed systems, and the infrastructure layer. CS:APP, Designing Data-Intensive Applications, and the databases section of the main curriculum are the natural continuation. Start building something that has real users, even if the number is small.

For CS students: algorithms at depth (CLRS cover to cover), discrete math (Lehman/Leighton/Meyer), and a rigorous probability text (Blitzstein & Hwang). The gap between a programmer and a computer scientist is mostly mathematical maturity — that maturity is acquired through doing problems, not reading solutions.

For Game Dev students: the Unity-specific skill curve continues, but the limiting factor past a certain point is usually not Unity knowledge — it's the underlying CS and math. Linear algebra (Strang) and the physics simulation basics that require it become relevant fast. So does architecture: games that grow past a prototype need the same structural discipline as any other software system.
